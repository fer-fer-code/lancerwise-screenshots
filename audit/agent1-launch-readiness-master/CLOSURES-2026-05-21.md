## Architectural insight — two patterns identified

### Pattern A — **Context Provider**

**When to use:** single-route widget-dense surfaces где many widgets sit на same page.

**Routes deployed:** /dashboard (#73, 21 widgets), /work/time + /time-tracker (#93, 80 widgets).

**Mechanism:**
- Single batched API endpoint (`/api/dashboard/widget-data`, `/api/time-tracker/widget-data`)
- React Context Provider mounted в (app)/layout.tsx, pathname-gated
- Widgets consume via `useDashboardData()` / `useTimeTrackerData()` hook
- Hook returns `null` during fetch → widgets show loading skeleton OR null guard

**Trade-offs:**
- ✅ Cross-widget data dedup (e.g. profile column subsets needed by 17 widgets → 1 fetch)
- ✅ Single fetch lifecycle, easy к reason about
- ❌ Over-fetches on subroute visits where shared data not needed
- ❌ Requires careful array-vs-object shape contract — Stage 1 v1 regression caught by [AGENT 3] probe (Provider returned objects, widgets expected arrays → WebKit + Chromium crash)

### Pattern B — **Server prefetch + initialProps**

**When to use:** subroute-heavy surfaces где each subroute renders а distinct widget cluster. ALSO: surfaces where WebKit hydration fan-out matters.

**Routes deployed:** /invoices/[id] (#74), /settings (#94 v2).

**Mechanism:**
- `page.tsx` async server component
- `await Promise.allSettled([N fetches])` resolves BEFORE first byte
- Data passed к а `'use client'` child component via typed props (`<RootClient profileRow={profileRow} servicePackages={...} />`)
- Each widget accepts `initial<X>?` prop, uses for initial state, skips own mount-fetch когда defined

**Trade-offs:**
- ✅ Data arrives BEFORE first render → eliminates hydration race
- ✅ WebKit hydration fan-out architecturally prevented (vs Pattern A's statistical reduction)
- ✅ Subroute-aware — only fetches what а subroute needs
- ✅ No new Context infrastructure к maintain
- ❌ Requires `'use client'` child extraction если page has searchParams / useTheme / useState
- ❌ Initial PR #132 (v1) tried в-place pattern WITHOUT server-component conversion → wiring race → REGRESSION (33 fetches vs 27 baseline)
- ✅ v2 PR #135 corrected к full server-component conversion → 2 fetches (PASS)

### Anti-pattern — Provider в `'use client'` page

**Insight discovered via PR #132 / PR #135 v1→v2 progression** + [AGENT 1] DISCOVERY.md analysis.

If you have а subroute-heavy surface like /settings, **do NOT migrate к Context Provider** like /work/time did. Pattern A's strength (cross-widget data sharing) becomes а liability when widgets are subroute-distributed. AND Provider в а `'use client'` page reintroduces hydration fan-out на WebKit.

**Correct pattern для /settings-like surfaces:** Server-component shell (Pattern B), NOT Provider.

## Lessons codified

### 1. Writer/verifier split is а structural QA win

Pattern proven repeatedly: [AGENT 2] writes fixes, [AGENT 3] writes probes, neither validates own work. Caught:

- Stage 1 v1 Chromium widget tree crash (#119 — would have shipped к users)
- Stage 2 PARTIAL verdict surfaced honestly (52 fetches, not claimed-target 35-45)
- #94 v1 REGRESSION (33 fetches FAIL — caught before re-deploy cascade)
- WebKit React #418 cross-route reconciled vs launch-baselines history

Every probe FAIL OR PARTIAL surfaced а real condition. Zero false-positives. Zero false-negatives. **Codify as permanent gate** для all future N+1 work и similar mount-fetch-pattern refactors.

### 2. Cross-engine probe для WebKit signals

Chromium-only testing would have missed:
- /settings v1 WebKit render-empty bug (existed pre-#94, not surfaced by Chromium probes)
- /settings v2 WebKit win preservation (only detectable by sampling WebKit bodyLen)
- Hydration fan-out's WebKit-specific failure mode

**Codify:** 4-cell × 3-run matrix (Chromium + WebKit × EN + RU) is the minimum probe coverage для any UI-mounting change. Single-engine probe = blind spot.

### 3. Sentry alone cannot catch React #418

React #418 (text-content hydration mismatch) fires в browser console. Sentry с PII scrub + standard config does NOT capture React.dev minified error codes as P1. Probe protocols caught these; Sentry did not.

**Codify:** depend on probe protocols, not Sentry, for hydration-class errors. Sentry's role = post-launch user-error reports, not pre-launch verification.

### 4. Hydration fan-out elimination = WebKit render fix

[AGENT 1] DISCOVERY.md identified mechanism: 27-widget simultaneous mount-fetches during hydration window caused React tree shape instability. WebKit (Safari/iOS) is stricter than Chromium и bails fatal via React #418 → tree unmount → bodyLen 246.

**Codify:** for any page rendering > ~10 widgets that each fetch on mount, default к Pattern B (server prefetch + initialProps). Pattern A acceptable only когда widget cluster is route-bound и Provider's data shape is а pure superset of every widget's needs.

### 5. Context Provider anti-pattern для /settings-like surfaces

Provider в а `'use client'` page that renders subroute-distributed widgets:
- Over-fetches when subroute не needs shared data
- Reintroduces hydration fan-out (Provider state transitions cause coordinated re-renders)
- Adds maintenance overhead (new Context к maintain)

**Codify:** before defaulting к Pattern A, check: (a) widgets all on same route, (b) shared data is superset of widget needs, (c) page can stay `'use client'` without subroute fragmentation. If any check fails → Pattern B.

### 6. Server-component conversion as fix-forward

v1 wiring race ([AGENT 2]'s analysis of PR #132) was nearly invisible — passed CI, caused regression only at runtime. Manual diagnosis would have taken hours. **[AGENT 3] post-deploy probe caught within minutes.**

Server-component conversion (v2, PR #135) фix-forward took ~2h focused work и architecturally eliminated the bug class. Future N+1 fixes на subroute-heavy surfaces should default к Pattern B from start.
